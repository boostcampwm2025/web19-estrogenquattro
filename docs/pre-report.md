# 온라인 모각코 서비스 기획서

## 🌱 잔디 심고 갈래?

> 개발자들을 위한 온라인 모각코(모여서 각자 코딩) 서비스

---

## 1. 기획 배경

### 혼자보다 함께할 때 힘을 얻은 적이 있나요?

저희 팀원들은 챌린지 과정 속에서 늦은 밤까지 작업할 때 슬랙 파란 불을 보며 힘을 얻었던 경험에 공감했습니다.

실제로는 곁에 없지만, **함께 코딩하고 있고 혼자가 아니라는 그 느낌**에 주목했습니다.

따라서 온라인에서 열정을 불태우고 있는 개발자들을 위해 **온라인 모각코 서비스**를 기획했습니다.

---

## 2. 서비스 목적

### 기존 서비스의 문제점과 해결 방향

| 기존 서비스 | 문제점과 한계 | 우리 서비스 |
|-------------|---------------|-------------|
| 온라인 모각코 | 진입장벽 존재 (접속 규칙, 화면/캠 공유) | 규칙 없이 즉시 참여 가능 |
| 메타버스 (ZEP, 게더타운) | 개발자 특화 기능 부재 | GitHub 연동으로 개발자 특화 기능 제공 |
| 디스코드/슬랙 | 공간감, 시각적 성취 부족 | 함께한다는 유대감, 시각적 성취 제공 |
| 오프라인 모각코 | 지리적 한계, 내향적인 성향 | 지리적 제한 없이 온라인으로 가볍게 접속 |

### 핵심 가치

- 개발자들에게 **'함께 코딩한다는 느낌'** 제공
  - 메타버스적 요소(캐릭터, 맵)를 통해 온라인에서 **서로를 인식하고 소속감 부여**
  - GitHub 활동(커밋, PR)과 같은 **코딩 과정을 시각적 이펙트로** 표현
  - 개인의 코딩 활동을 모아 맵을 변화(황무지→숲)시키는 **연대 경험** 제공

### 리텐션 전략

- **게이미피케이션 요소 강화**: 개인 작업에 방해되지 않는 선에서 시각적 흥미 요소 추가 (펫, 마스코트 등)
- **운영 경험 중시**: 기술적 복잡도보다는 실제 서비스를 운영하며 겪는 트래픽 처리와 리텐션 유지에 집중

### 타겟 유저

- 늦은 시간 홀로 작업하면서 동기부여를 원하는 개발자
- 주변 지역에 동료가 없는 개발자
- 성장하고 싶지만 내향적인 성격을 가진 개발자

## 3. 서비스 기능

### 3.1 주요 기능 개요

| 기능 | 목적 | 도입 이유 |
|------|------|----------|
| 변화하는 가상 공간 | 함께 몰입하는 공동 작업 환경 제공 | 같은 공간에 머물며 함께 행동하고, 그 결과로 공간을 변화시키는 경험 제공 |
| 캐릭터 시스템 | 사용자 아바타 및 실시간 움직임 | 언어 없이도 존재감과 동질감 형성 |
| GitHub 활동 연동 | 커밋, PR 등 개발 활동 추적 | 개발자 특화 동기부여 수단 |
| 맵 성장 시스템 | 전체 맵이 커밋 수에 따라 변화 | 개인 성취를 공동의 성과로 확장 |
| 개인 성취 시각화 | 기여도, 집중 시간 등 표시 | 개인의 노력 가시화 및 지속 동기 제공 (w05-day04 추가) |
| 시즌제 운영 | 시즌별 맵 리셋/변화 | 장기적 참여 유도 및 목표 재설정 |

### 3.1-1 기능 우선순위 보완

- GitHub 연동은 핵심 기능보다 **보조적 요소로 활용**하는 방향 검토 (w05-day02 추가)

### 3.2 변화하는 가상 공간 (맵 성장 기반 메타버스)

사용자는 서비스 접속 시, **아직 아무것도 개척되지 않은 '황무지' 상태의 가상 공간**에 입장합니다.

이 공간은 단순한 배경이 아니라, **사용자들의 개발 활동에 따라 점진적으로 변화하는 공동의 세계**입니다.

#### 맵 및 게이지 시스템

- **게이지 공유**: 각 방(채널)의 게이지는 개별이 아닌 **전체 공유**되어 계산
- **오브젝트 상호작용**: 사용자 캐릭터는 오브젝트를 통과할 수 없으며, 화면 크기에 맞춰 오브젝트 크기 조정
- **랜덤 방 배정**: 접속 시 랜덤 방 배정, 초기 방 3개/방당 최대 14명 기준 (w05-day02 추가). 방 5개 고정으로 변경, 동적 생성 대신 고정 채널 방식 채택. 채널 기능 도입 전까지 랜덤 배정 비활성화(한 채널 집중) (w09-day02 3차 수정)
- 랜덤 입장으로 작업 진입 허들 감소 (w05-day05 추가)
- 데모 시 단일 룸 고정 적용 (w05-day05 추가)
- 줌아웃 제한: 맵 전경이 보이는 범위까지만 허용, 필요 시 줌 고정 검토 (w06-day04 추가)
- 맵 스테이지 이미지 확장: 신규 맵 이미지 적용 및 장애물 처리 후 단계 확장 (w06-day04 추가)
- 맵 이미지 버전: 스테이지 3, 4 새 버전 적용 (w08-day03 추가)
- 맵 변경 시 리스폰: 맵 스테이지 변경 시 제자리 유지 확정, 고립 발생 시 새로고침으로 해결 가능 (w07-day02 추가, w09-day02 2차 확정)
- 맵 전환 포인트 차등 설정: 단계별 200/300/400/500 포인트로 설정, 초반 빠른 성장 유도 (w09-day01 추가)
- 맵 최대 단계 순환 방지: 5단계 도달 후 게이지 초과 시 더 이상 전환하지 않음 (w09-day01 추가)
- 방 5개 고정 + 채널 선택 UI: 초기 입장은 랜덤 배정 유지(꽉 차면 선형 탐색), 이후 사용자가 채널(방)을 직접 선택하여 이동 가능. 5개 채널 스크롤 없이 한 번에 표시, 현재 채널 번호·인원수(현재/최대)·입장 가능 여부 표시. 방 정보 API로 입장 가능 방 목록/최대 인원/현재 접속 인원 반환. 방 전환 시 예약-확인 패턴 적용(예약 후 이동, 30초 타임아웃으로 유령 예약 방지, 동시 경합 시 후발 요청 거부). 채널 변경 시 음악 강제 중지·타이머 초기화 확정(disconnect가 새로고침/채널이동 미구분), 아이리스 트랜지션 전환 효과 검토 (w09-day01 2차 추가, w09-day02 3차 확정)
- 채널 선택창 새로고침 버튼: 방 현황 실시간 조회용 수동 새로고침 버튼 추가, 폴링 없이 사용자 주도로 방 정보 갱신 (w09-day03 추가)
- 첫 접속 채널 배정 방식: 차곡차곡(인원 많은 채널 우선) 배정 확정, 혼자 배정 시 이탈 방지. 추후 점진적 개방 알고리즘(방 2→3→4개 순차 개방) 검토 (w09-day03 보류, w09-day03 2차 확정)

#### Stage 1: 황무지 (초기 화면)

- 맵은 메말라 있고, 비어 있는 상태로 시작
- 사용자는 이 공간 안에 스폰되어 다른 사용자들의 존재를 확인 가능
- 화면 최상단에는 **글로벌 진행 바**가 고정으로 표시
- 공통 미션은 **커밋 개수**, **집중 시간** 등 다양한 요소를 조합

#### Stage 2: 숲 (성장 화면)

- 공통 미션 달성 시, 황무지가 점진적으로 **숲으로 변화**
- 메마른 지형에 초목이 자라나고, 오브젝트와 색감이 추가
- **해당 공간에 접속 중인 모든 사용자에게 동시에 반영**
- 핵심: **완성 자체가 아니라 성장 중인 상태를 함께 바라보는 경험**

#### 맵 컨셉 (시즌별 확장 가능)

- 황무지 → 숲 (개척 컨셉)
- 허름한 카페 → 초호화 카페
- 디스토피아 → 유토피아
- 폐허 → 활기찬 마을

#### 시즌제 운영 정책

- **기간**: 시즌 초기화 기간은 **7일** 단위로 설정, 월요일 자정 기준 주간 리더보드 방식 (w08-day04 확정)
- **시즌 변경**: 시즌이 변경될 때 맵의 테마 변경
- **게이지 만료**: 게이지 바가 가득 찼을 때, 다른 맵 이미지로 변경
- 맵 전환 동기화: 서버에서 인덱스 브로드캐스트하여 클라이언트 동기화 (w08-day03 추가)

### 3.3 개인 성취 시각화

- 타 사용자와의 직접적인 순위 경쟁보다는, **"나는 오늘 이만큼 했다"**는 자기 확인에 초점
- 비교로 인한 스트레스가 아닌 **자기 몰입형 동기부여**
- 공동 목표뿐 아니라 **개인 목표에도 집중**하는 방향 검토
- 개인 성취와 공동 맵 성장을 병행하는 **투 트랙 구성** 검토 (w05-day02 추가)

## 4. 사용자 시나리오

```
1. 사용자는 GitHub 소셜 로그인을 통해 서비스에 접속
2. 랜덤 닉네임(또는 GitHub 닉네임)으로 가상 공간에 스폰
3. 캐릭터 머리 위에 다른 사용자들의 GitHub 네임과 집중 시간 확인
4. 사용자는 자리를 잡고 캐릭터를 이동한 뒤 개인 작업 시작
5. 로컬 환경에서 개발 후 git push 수행
6. 커밋 이벤트 감지 → 캐릭터 이펙트 발생 + 글로벌 게이지 상승
7. 목표 달성 시 맵이 변화, 사용자들은 이모지 리액션으로 반응
```

- SQLite3 선택 이유: 1GB 메모리 환경에서 파일 기반 DB로 메모리 절약, 성능은 낮지만 무료 서버 제약 대응 (w06-day04 추가)

- 엔진엑스 도입: 인증서 자동 갱신, 메모리 효율성(15~22MB), epoll 방식 IO 성능 우수 (w07-day04 추가)
- 모니터링 툴: Netdata 메모리 소모량 과다로 사용 불가, 오라클 클라우드 기본 모니터링 또는 Sentry 검토 (w07-day04 추가)
- 로깅 시스템 개선: 외부 공개 대비 에러 로그 분리 저장 및 조회 가능한 시스템 구축 필요, 불필요한 로그로 인한 부하 정리 포함. Axiom + Winston 어댑터 확정(Pino 대신 기존 Winston 유지), 구조화 JSON 로깅으로 전환, 월 500GB 무료 (w08-day05 추가, w09-day01 구체화, w09-day02 확정)
- 마이그레이션 롤백 메커니즘: migration run 실패 시 SQLite 롤백 여부 확인 필요, 데이터 오염 방지 검토. 현재 서버 단독 마이그레이션 방식 시도 중 (w08-day05 추가)
- 무중단 배포: PM2 클러스터 모드는 인메모리/소켓/폴링 제약으로 어려움. 웹소켓+인메모리 구조 특성상 Redis 등 상태 중앙화 없이 완전 무중단 구현은 한계가 있어, 차선책으로 클라이언트 소켓 재연결 로직(5~10분간 재시도) + 서버 갈아끼우는 시간 최소화 방식을 유지 (w09-day01 2차 추가, w09-day02 구체화, w11-day05 재확인)
- 머지 운영 원칙: 무중단 배포 적용 전에는 대규모 머지를 자제하고, PR은 `dev` 브랜치로 먼저 통합한 뒤 `main`으로 일괄 반영하여 배포 이벤트를 줄이는 방식으로 운영 (w10-day05 2차 추가, w11-day05 수정)
- 프론트엔드 빌드 통일 검토: 테스트 라이브러리 미설치로 정적 빌드 실패 사례 발생, 정적 빌드 통일 논의 중 (HMR 미지원 단점으로 보류) (w09-day01 2차 추가)
- GCP 마이그레이션: 해외 공개 대응을 위해 오라클 서버에서 GCP로 이전하기로 결정. 단기 공개 일정(2/13) 우선으로 SQLite/현행 배포 구조를 유지하고 대규모 DB 전환은 보류 (w10-day02 추가)
- GCP 초기 리전/사양: 초기 배포는 한국 리전 우선으로 적용하고, SQLite 유지 구조 특성을 고려해 CPU 우선 상위 사양(월 약 $94 수준)으로 운영 시작 (w10-day03 추가)
- 인프라 전환 실행: 오라클 서버를 종료하고 GCP 단일 운영으로 전환 완료. HTTPS는 신규 도메인 확정 후 적용 (w10-day04 추가)
- 도메인 전환 이슈 대응: `commitplease.com` 적용 직후 일부 로그인 루프/도메인 튕김 이슈가 발생해 CORS·환경변수·Nginx·캐시를 교차 점검. 브라우저 캐시 무효화 및 설정 재배포로 안정화 (w10-day04 2차 추가)
- 도메인 병행 운영: 기존 `jandiplease.duckdns.org`는 즉시 폐기하지 않고 유입 보존용으로 유지하며, 신규 도메인 중심으로 점진 전환 (w10-day04 2차 추가)
